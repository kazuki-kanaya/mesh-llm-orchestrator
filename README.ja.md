# mesh-llm-orchestrator

Mesh LLM をオンプレミスの LLM 推論サーバーとして非同期的に安定稼働させるための基盤。

透過性を保つことで、基本的には Mesh LLM に限らず任意の HTTP API に対しても互換的に適用できる構造を目指す。

---

## ステータス

* 現在は実装初期段階
* リポジトリには設計文書とバックエンドの初期実装が含まれる
* 現時点では Redis を中心にした非同期実行基盤として設計している

---

## このリポジトリが目指すもの

* 任意の HTTP リクエストをジョブとして受け付け、非同期実行できるようにする
* リクエストおよびレスポンスを透過的に扱う
* Queue の不完全性を前提にしつつ、状態管理によって整合性を保つ
* オンプレミス環境でも水平スケール可能な実行基盤を構築する

---

## 概要

本システムは、任意の HTTP API リクエストを即時実行するのではなく、ジョブとして受け付け、後続の Worker が非同期に実行する基盤である。

リクエストおよびレスポンスは内容を解釈せず、そのまま保存・転送する。JSON パースやフィールド変換を前提とせず、HTTP ヘッダも含めて透過的に扱う。

配送手段としての Queue は不完全であることを前提とし、状態管理および整合性維持の中心には Redis を置く。

---

## 目的

### 非同期 HTTP 実行基盤の構築

同期的な HTTP 実行では扱いづらい長時間処理や高負荷処理を、ジョブとして安全に実行できるようにする。

---

### 透過性の維持

* リクエスト本文はバイト列としてそのまま扱う
* レスポンス本文も同様にそのまま保存・返却する
* アプリケーション固有の JSON スキーマや業務ロジックに依存しない

---

### 分散前提での整合性確保

* Worker のクラッシュ
* Queue の重複配信
* ネットワーク断

を前提にし、それらを完全に防ぐのではなく、状態管理によって最終的な整合性を保つ。

---

### スケーラビリティの確保

* API 受付
* 非同期実行
* Worker 数

を将来的に Kubernetes 上で水平スケールできる構造にする。

## 想定ユースケース

* クライアントが長時間処理を伴う HTTP リクエストを送る
* サーバは即時に `job_id` を返し、後から結果を取得させる
* 複数 Worker が同時にジョブを処理する
* Queue の重複配信が起きても、状態管理で最終結果を一貫させる

---

## 設計

### 全体構成

```text
Client
  ↓
API Gateway
  ↓
Orchestrator
  ↓
Queue (Redis List)
  ↓
Executor (Worker x N)
  ↓
Redis (state / request / response / lock)
```

---

### レイヤ構造

#### 1. API Gateway

役割:

* クライアントからの HTTP リクエスト受付
* ジョブ作成 API の提供
* ジョブ状態問い合わせ API の提供
* 結果取得 API の提供

---

#### 2. Orchestrator

役割:

* `job_id` の生成
* リクエストの保存
* 初期状態 `queued` の設定
* Queue への投入
* クライアントへの `job_id` 返却

特徴:

* Queue の完全性に依存せず、Redis の状態を正とする

---

#### 3. Queue

役割:

* `job_id` の配送

特徴:

* Redis List を利用する
* 重複配信や順序崩れを許容する
* 信頼性の中心ではなく、あくまで配送手段として扱う

---

#### 4. Executor

役割:

* Queue から `job_id` を取得
* Redis ロックを取得
* 状態を `running` に更新
* 保存された HTTP リクエストを取得
* 外部 API に透過的に送信
* レスポンスを保存
* 状態を `completed` または `failed` に更新

---

#### 5. Redis

役割:

* 状態管理
* リクエスト保存
* レスポンス保存
* ロック管理
* Queue
* Pub/Sub

特徴:

* 本システムにおける唯一の真実として扱う

---

### コンポーネント責務

| コンポーネント | 役割 |
| --- | --- |
| API Gateway | 外部 API 提供、ジョブ状態問い合わせ |
| Orchestrator | ジョブ生成、初期状態設定、Queue投入 |
| Queue | `job_id` 配送 |
| Executor | HTTP 実行、結果保存、状態更新 |
| Redis | 状態・データ・ロックの一元管理 |

---

### 実行フロー

#### 1. ジョブ作成

1. クライアントが HTTP リクエストを送信する
2. Orchestrator が `job_id` を生成する
3. リクエストをそのまま保存する
4. `status = queued` を設定する
5. Queue に `job_id` を投入する
6. クライアントへ `job_id` を返却する

---

#### 2. ジョブ実行

1. Executor が Queue から `job_id` を取得する
2. `SETNX` によりロックを取得する
3. ロック取得失敗時はスキップする
4. `status = running` を設定する
5. 保存済みリクエストを取得する
6. 外部 API に透過的にリクエスト送信する

---

#### 3. 完了処理

1. レスポンスを受け取る
2. レスポンスを保存する
3. `status = completed` を設定する

失敗時は `status = failed` を設定する。

---

#### 4. 結果取得

1. クライアントが `job_id` を指定して問い合わせる
2. API Gateway または Orchestrator が Redis から状態を取得する
3. `completed` であればレスポンスを返す

---

### 状態遷移

```text
queued -> running -> completed
                 \-> failed
```

---

### 排他制御

```text
SETNX(lock:job:{id})
```

* 最初にロックを取得した Executor のみが実行する
* ロックは TTL 付きとする

---

### 整合性モデル

#### at-least-once

同一 job が複数回実行される可能性を許容する。

---

#### first-writer-wins

最初に `completed` を確定した結果を有効とし、後続結果は無視する。

---

### 再実行

`running` のまま一定時間を超えて停滞したジョブは stale とみなし、必要に応じて `queued` へ戻す。

---

### TTL ポリシー

* `completed` は一定時間後に削除する
* `failed` も一定時間後に削除する

---

### ストリーミング拡張

将来的には以下を行える構造を残す。

* Executor がレスポンスを逐次 Pub/Sub で配信
* Orchestrator が `job_id` 単位で購読
* クライアントへリアルタイム転送

---

## 設計方針

### 1. 透過性を維持する

リクエストとレスポンスは解釈せず、バイト列として扱う。

---

### 2. Queue を信用しない

Queue は配送手段として使うが、最終状態の正しさは Redis の状態で判断する。

---

### 3. Redis を唯一の真実とする

ジョブ状態、実行済みかどうか、レスポンス、ロックなどの管理は Redis に集約する。

---

### 4. 実行制御はロックで行う

実行の排他は `SETNX` によって担保する。

---

### 5. レスポンスは Queue に戻さない

結果は Redis に保存し、問い合わせ時に参照する。

---

## まとめ

本システムは、不完全な Queue を用いて処理を流しつつ、Redis 上の状態管理によって最終的な整合性を保つ非同期 HTTP 実行基盤である。

透過性、非同期性、分散実行、再実行可能性を重視し、オンプレミス環境でも運用可能なスケーラブルな構造を目指す。
